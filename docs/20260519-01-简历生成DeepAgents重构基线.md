# 简历生成 Deep Agents 重构基线

更新时间：2026-05-19

## 1. 文档目的

本文档记录「简历生成」功能当前的重构状态、端到端流程、关键实现文件、已解决的问题、仍需处理的风险和后续优化方向。后续继续优化该功能时，应以本文档作为技术基线，避免重复讨论已经确定的约束和方案。

## 2. 当前目标

简历生成希望从“基于项目元数据/提交统计的简单生成”升级为“基于项目代码理解的生成”：

- 用户选择书架中的一个或多个项目。
- 系统为每个项目生成一份可编辑、可版本化的「项目背景知识」。
- 系统基于背景知识、岗位方向、JD 关键词和语气偏好，生成 STAR 结构的项目经历。
- 用户可以编辑项目经历，并导出 Markdown / Word。

核心体验要求：

- 用户侧保持一键操作，不增加手动选择文件、分步确认等负担。
- 项目代码探索必须可靠，不能在 Agent 递归失败后静默使用半截结果。
- DeepSeek 当前是主要验证模型，需要兼容其 OpenAI-compatible API 差异。

## 3. 当前实现结论

本轮重构后，「背景知识生成」不再使用 Deep Agent 自由探索项目文件，而是改成受控的确定性流程：

1. 后端基于项目根目录生成完整可见文件索引。
2. 模型只根据索引规划需要读取哪些关键文件。
3. 程序按计划读取文件内容。
4. 模型根据索引和已读文件合成背景知识 Markdown。

这个调整的原因是：Deep Agent 自由调用 `project_list_dir` / `project_read_file` / `project_grep` 时容易出现工具循环，最终触发 LangGraph 的 `GRAPH_RECURSION_LIMIT`。用户明确要求不能因为递归限制而停掉探索或接受部分结果，所以背景知识阶段必须改为有边界、可解释、可失败的流程。

「简历项目经历生成」目前仍使用 Deep Agent，但只在虚拟文件系统中读取已有背景知识并写出 `/resume.json`，不再直接探索真实项目目录，风险明显更低。

## 4. 功能入口与页面流程

入口：

- 页面：`src/pages/Toolbox/index.tsx`
- 功能面板：`src/pages/Toolbox/ResumeGenerator/index.tsx`

用户流程：

1. 进入「工具箱」中的「简历生成」。
2. 在「选项目」页选择一个或多个书架项目。
3. 确认默认 AI 供应商可用。
4. 进入「背景知识」页，对每个项目点击「生成」或「重新生成」。
5. 进入「岗位配置」页选择岗位方向、语气，并可填写 JD 关键词。
6. 进入「生成简历」页点击「生成简历」。
7. 对生成结果进行编辑、保存、导出 Markdown / Word。

当前标签页：

- `select`：选择项目。
- `knowledge`：生成、编辑、查看、恢复项目背景知识。
- `resume`：基于背景知识生成 STAR 项目经历并导出。

## 5. 背景知识生成流程

入口文件：

- `src/pages/Toolbox/ResumeGenerator/KnowledgePanel.tsx`
- `src/services/resume/agents/knowledgeAgent.ts`
- `src/services/resume/tools/projectIndex.ts`
- `src-tauri/src/commands/resume.rs`

端到端流程：

1. 用户点击背景知识页右上角「生成」。
2. `KnowledgePanel` 调用 `runKnowledgeAgent`。
3. `runKnowledgeAgent` 调用后端 `resume_project_index`。
4. Rust 后端读取项目路径，校验项目是否存在，应用 ignore 规则并生成索引。
5. 前端将项目基本信息、索引统计、一级目录、候选关键文件发送给模型。
6. 模型返回严格 JSON 格式的读取计划：

```json
{
  "files": ["README.md", "package.json", "src/main.tsx"],
  "reasons": {
    "README.md": "项目概览",
    "package.json": "依赖和技术栈"
  }
}
```

7. 前端校验模型计划：

- 文件必须存在于后端索引中。
- 路径不能越界。
- 最多读取 `45` 个文件。
- 如果模型返回空计划，则用程序内置评分规则生成 fallback 计划。

8. 前端逐个调用后端 `resume_project_read_file` 读取文件内容。
9. 前端将索引摘要、现有背景知识、已读文件内容发送给模型。
10. 模型输出完整 Markdown 背景知识。
11. `KnowledgePanel` 将结果保存到 `resume_knowledge/<projectId>.md`，旧版本自动备份到历史目录。

背景知识固定章节：

- 项目概览
- 技术栈详情
- 核心功能模块
- 架构亮点
- 可挂载 JD 关键词

当前限制：

- 单文件读取由后端限制为 `10000` 字符。
- 模型上下文拼接由前端限制为 `120000` 字符。
- 索引最多允许 `20000` 个可见文件，超过时明确报错，要求用户用 `.codeshelfignore` 排除无关目录。
- 索引深度当前为 `16`。

## 6. 项目文件访问设计

项目文件读取不再由前端 Tauri fs 插件直接访问，而是通过 Rust 命令访问：

- `resume_project_index`
- `resume_project_list_dir`
- `resume_project_read_file`
- `resume_project_grep`

后端约束：

- `project_id` 必须来自 SQLite `projects` 表。
- 项目根目录会 canonicalize。
- 相对路径会做路径穿越校验。
- 被 ignore 的文件/目录不可读。
- ignore 来源包括内置规则、`.gitignore`、`.codeshelfignore`。

这样做的收益：

- 避免 Tauri `fs:scope` 对动态项目路径的限制。
- 文件读取、搜索和索引规则集中在后端，行为更一致。
- 后续可以在 Rust 层增加敏感文件拦截、二进制识别、缓存和大项目扫描优化。

## 7. 模型调用设计

入口文件：

- `src/services/resume/llmFactory.ts`
- `src/services/resume/llmProxyFetch.ts`
- `src-tauri/src/commands/resume.rs` 中的 `llm_proxy_request`

当前模型调用路径：

1. 前端使用 LangChain `ChatOpenAI`。
2. `ChatOpenAI` 注入自定义 `fetch: tauriLlmFetch`。
3. `tauriLlmFetch` 将请求转发给 Tauri 命令 `llm_proxy_request`。
4. Rust 后端使用 `reqwest` 调用模型供应商 API。
5. 响应返回前端，交给 LangChain 继续处理。

当前不是流式输出：

- `streaming: false`
- `streamUsage: false`
- `disableStreaming: true`
- 代理层强制 `stream = false`
- 删除 `stream_options`

DeepSeek 兼容处理：

- 如果 provider 是 DeepSeek 或 baseURL 包含 `deepseek`：
  - 请求体设置 `thinking: { "type": "disabled" }`
  - 删除 `enable_thinking`
  - 删除 `reasoning`
  - 删除 `reasoning_effort`
  - 删除 `reasoning_content`
  - 设置 `parallel_tool_calls: false`
- 非 DeepSeek：
  - 设置 `enable_thinking: false`
  - 删除 `thinking`
  - 设置 `parallel_tool_calls: false`

做这个处理的原因：

- DeepSeek thinking 模式会返回 `reasoning_content`。
- 多轮工具调用时，DeepSeek 要求把 `reasoning_content` 回传，否则会报错。
- LangChain / OpenAI-compatible 消息链路当前不能可靠保留并回传该字段。
- 所以 Resume 相关流程暂时禁用 thinking，并使用非流式直出。

## 8. 简历项目经历生成流程

入口文件：

- `src/pages/Toolbox/ResumeGenerator/ResumePanelV2.tsx`
- `src/services/resume/agents/resumeAgent.ts`

流程：

1. 用户在「生成简历」页点击生成。
2. `ResumePanelV2` 调用 `runResumeAgent`。
3. `runResumeAgent` 将每个项目背景知识放入 Deep Agent 虚拟文件系统：

```text
/knowledge/<projectId>.md
```

4. Deep Agent 根据岗位方向、JD 关键词、语气偏好读取背景知识。
5. Deep Agent 将结果写入：

```text
/resume.json
```

6. 前端解析 `/resume.json`，转换为 `ResumeV2`。
7. 页面展示项目经历、技能词云、个人简介。
8. 用户可以编辑单条项目经历。
9. 用户可以保存或导出 Markdown / Word。

当前 Resume Agent 输出结构：

```json
{
  "summary": "一句话个人简介",
  "skills": ["TypeScript", "React"],
  "experiences": [
    {
      "projectId": "项目 id",
      "projectName": "项目名",
      "techStack": ["技术栈"],
      "star": {
        "situation": "项目背景",
        "task": "承担任务",
        "action": "技术行动",
        "result": "项目成果"
      }
    }
  ]
}
```

约束：

- 所有内容必须基于背景知识。
- 禁止编造项目细节、人数、时间、指标。
- JD 关键词只有在背景知识中真实存在对应技术时才能命中。
- 项目顺序与用户选择顺序一致。

## 10. 已解决的问题

### 10.1 Tauri 前端无法直接读取项目任意路径

原因：项目路径是用户动态添加的本地目录，Tauri fs 插件 scope 不适合直接覆盖全部项目路径。

处理：文件访问改为 Rust 命令，后端从项目表校验根目录后读取。

### 10.2 Deep Agents 浏览器打包报错

报错：

```text
Uncaught TypeError: Cannot read properties of null (reading 'constants')
```

原因：`deepagents` 初始化时访问 Node `fs.constants.O_NOFOLLOW`。

处理：在 Vite 中增加 `node:fs` shim，使前端打包可运行。

### 10.3 DeepSeek thinking 相关 400 错误

典型错误：

```text
The `reasoning_content` in the thinking mode must be passed back to the API.
```

```text
thinking: invalid type: boolean false, expected struct ThinkingOptions
```

处理：

- DeepSeek 使用 `thinking: { "type": "disabled" }`。
- 不再发送 boolean `thinking: false`。
- 移除 `reasoning_content` 等字段。
- Resume 相关流程禁用 thinking。

### 10.4 LangGraph 递归上限

报错：

```text
Recursion limit of 60 reached without hitting a stop condition.
```

原因：背景知识阶段让 Deep Agent 自由探索真实项目文件，容易陷入“继续列目录/继续读文件”的循环。

处理：背景知识阶段改为确定性管线，不再使用自由工具循环。

## 11. 当前风险和待补齐事项

### P0：需要优先处理

1. 修复 `src/bindings.ts` 生成问题

当前 `npm run build` 会失败：

```text
Duplicate identifier 'ShortcutInput'
TAURI_CHANNEL is declared but its value is never read
__makeEvents__ is declared but its value is never read
```

这不是简历功能本次改动引入的，但会影响完整构建链路。需要检查 specta 导出、重复类型来源和 tsconfig 对 generated bindings 的检查策略。

2. 增加背景知识生成的取消控制

`runKnowledgeAgent` 已预留 `AbortSignal`，但 UI 当前没有暴露取消按钮。大项目或慢模型下用户只能等待。

建议：

- `KnowledgePanel` 增加取消按钮。
- 每次生成创建 `AbortController`。
- 取消时停止后续文件读取和模型调用。

3. 给背景知识流程增加更明确的阶段状态

当前 UI 仍显示“Agent 正在探索项目代码...”，但新流程已经不是自由探索。

建议改为：

- 建立项目索引
- 规划关键文件
- 读取核心文件
- 生成背景知识
- 保存版本

### P1：高价值优化

1. 持久化背景知识生成流程明细

当前背景知识生成过程中只在 UI 上简单展示步骤，刷新页面或生成结束后无法回看完整执行链路。

建议新增“生成记录”：

- 每次生成创建一条 `knowledge_run` 记录，关联 `projectId`、开始时间、结束时间、状态、模型供应商、模型名。
- 保存每一步的结构化明细，例如：
  - 阶段：建立索引、规划文件、读取文件、补充读取、生成背景知识、保存版本。
  - 模型请求摘要：system prompt 版本、user prompt 摘要、输入 token 估算。
  - 工具/后端调用：调用了什么命令或工具、参数是什么。
  - 文件读取：读取了哪个文件、读取原因、返回字符数、是否截断、是否被 ignore 或敏感规则拦截。
  - 模型输出：读取计划 JSON、最终背景知识摘要、错误信息。
- UI 增加“查看生成过程”入口，可在背景知识详情里展开查看历史生成记录。
- 生成失败时也必须保存失败记录，便于排查。

注意事项：

- 不能把完整敏感文件内容写入日志。
- 文件内容返回值建议只保存摘要、前后少量片段或 hash；完整内容是否保存需要单独设计权限和存储策略。
- 日志结构应服务于后续调试和用户解释，而不是只保存纯文本流水。

2. 背景知识系统提示词可编辑

当前背景知识生成的系统提示词写死在代码中，用户无法根据自己的简历风格、岗位方向或项目类型调整生成标准。

建议：

- 在简历生成器设置中提供“背景知识系统提示词”编辑入口。
- 提供默认模板、恢复默认、保存自定义模板。
- 模板中支持变量占位符，例如：
  - `{projectName}`
  - `{projectPath}`
  - `{projectTags}`
  - `{projectLabels}`
  - `{indexSummary}`
  - `{existingBackground}`
  - `{fileContext}`
- 自定义提示词保存到应用配置或单独的 resume prompt 配置文件。
- 每次生成记录中保存当次使用的提示词版本或快照，方便复现。

需要约束：

- 即使用户修改提示词，也要保留硬性安全规则：禁止编造、敏感文件不发送、量化指标必须有来源。
- 提示词编辑区应有“默认提示词预览”和“变量说明”。

3. 简历生成系统提示词可编辑

当前 `ResumeAgent` 的系统提示词固定在代码中，用户无法针对不同岗位、公司风格或简历表达偏好做深度调整。

建议：

- 在简历生成页或设置页提供“简历生成系统提示词”编辑入口。
- 支持默认模板、恢复默认、保存自定义模板。
- 模板支持变量占位符：
  - `{jobDirection}`
  - `{jdKeywords}`
  - `{tone}`
  - `{knowledgeDocs}`
  - `{projectOrder}`
- 自定义提示词应与岗位方向、语气偏好共同生效。
- 每份保存的简历可记录当次使用的提示词版本或快照。

需要约束：

- schema 输出要求不能被用户提示词破坏，仍必须产出可解析的 `ResumeV2` / JSON。
- 禁止编造的硬性规则应作为系统级 guardrail 保留。

4. 增加“补充读取”阶段

当前流程只有一次读计划。模型读完第一批文件后，可能发现还需要补充文件。

建议增加最多 1 轮补充：

- 模型基于已读内容和索引提出 `additionalFiles`。
- 程序校验并读取。
- 再进入最终总结。

需要保持边界：

- 最多补充 10-15 个文件。
- 不能无限循环。
- 补充理由必须写入步骤日志，方便用户理解。

5. 增强项目索引质量

当前索引主要包括路径、大小、扩展名、目录列表。

建议增加：

- `keyFiles`：README、依赖文件、入口配置文件。
- `moduleCandidates`：按目录聚合的候选模块。
- `extensionStats`：后端直接统计扩展名。
- `largeFiles`：大文件列表。
- `ignoredSummary`：被内置规则忽略的顶层目录提示。

6. 改进候选文件评分规则

当前评分规则在前端 `knowledgeAgent.ts` 中硬编码。

建议：

- 抽到独立模块，便于测试。
- 按技术栈拆分策略：Node、Java、Rust、Go、Python、Tauri 等。
- 对 monorepo、前后端分离项目增加更准确识别。

7. 增加敏感文件保护

当前内置 ignore 包括常见构建产物和 lock/min/map 文件，但敏感文件策略还不完整。

建议默认拦截：

- `.env`
- `.env.*`
- `*.pem`
- `*.key`
- `id_rsa`
- `id_ed25519`
- `secrets.*`
- `credentials.*`
- 含 token/password/secret/key 的配置文件

注意：敏感文件应默认不发送给模型，但可以在 UI 中提示“已跳过敏感文件”。

8. 背景知识增加证据链

当前背景知识会列关键文件，但没有结构化证据。

建议在文档末尾增加：

```markdown
## 证据来源
- README.md：项目概览
- package.json：技术栈
- src/pages/...：核心功能模块
```

这样后续简历生成可以更稳，也方便用户判断是否可信。

9. Resume Agent 也改为非 Deep Agent 直接 JSON 生成

当前 Resume Agent 使用 Deep Agent 读虚拟 FS 并写 `/resume.json`。风险较背景知识低，但仍有递归限制和虚拟文件系统依赖。

可选优化：

- 直接用 `ChatOpenAI.invoke` 生成 JSON。
- 使用 zod/schema 做输出校验。
- JSON 不合法时自动进行一次“修复 JSON”模型调用。

### P2：体验和质量优化

1. 支持背景知识批量生成

当前背景知识逐项目生成，用户选择多个项目时需要逐个点击。

建议：

- 在背景知识页增加“批量生成选中项目背景知识”按钮。
- 对选中项目顺序执行，避免同时占用模型和文件扫描资源。
- 展示批量进度：总数、当前项目、成功数、失败数、跳过数。
- 单个项目失败不阻断后续项目，但最终汇总失败列表。
- 支持“仅生成缺失背景知识”和“重新生成全部已选项目”两种模式。
- 批量生成的每个项目都应保存独立生成流程记录。

2. 历史简历名称修改功能

当前历史简历列表主要展示岗位方向、项目数量和更新时间，用户难以区分多个版本。

建议：

- `ResumeV2` 增加可选字段 `title` 或 `name`。
- 新生成简历默认名称可按规则生成，例如：
  - `{岗位方向}简历 - {项目数}个项目 - {日期}`

3. 增加生成质量检查

- 是否包含固定章节。

## 15. 重要设计原则

后续优化时建议继续遵守：

- 用户侧保持一键生成，不把技术复杂度转嫁给用户。
- Agent 不直接无限探索真实文件系统。
- 所有文件读取必须经过后端项目根目录校验。
- 不能在模型失败、递归失败或读取不完整时静默产出“看起来正常”的结果。
- 生成内容必须有项目文件依据，尤其是技术栈和量化指标。
- DeepSeek thinking 模式在工具/多轮场景中暂不启用，除非能完整保存并回传 `reasoning_content`。
