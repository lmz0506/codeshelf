# CodeShelf AI 供应商模块 PRD

## 1. 文档信息

- 文档编号：20260318-01
- 文档名称：AI 供应商模块 PRD
- 适用版本：待开发版本
- 关联需求：GitHub Issue #31（设置里新增 AI 设置）
- 关联上游需求：GitHub Issue #21（AI 自动生成简历项目经历）
- 关联背景：为产品提供统一的 OpenAI 兼容 AI 厂商接入能力，作为产品级 AI 基础设施，服务后续对话页、简历生成等 AI 能力使用

---

## 2. 需求背景

当前产品缺少统一的 AI 厂商配置能力，无法让用户在应用内灵活接入不同 AI 提供商并进行对话使用，也无法为后续 AI 简历生成等功能提供统一模型来源。

结合 GitHub Issue #21 的业务背景，本模块并不是单纯服务一个对话页，而是需要作为产品内 AI 能力的通用基础设施存在。后续类似“基于项目 README、commit、依赖文件生成简历项目经历”的能力，也需要复用本模块提供的模型配置与调用能力。

本次需求需要在设置中新增 AI 供应商模块，提供一套通用的 AI 配置机制：

1. 内置主流 AI 厂商模板；
2. 支持用户自定义兼容 OpenAI 协议的厂商；
3. 支持每个厂商配置多个模型；
4. 支持为模型配置 `thinking` 开关；
5. 支持设置默认模型；
6. 支持设置全局默认供应商；
7. 为对话页面、简历生成及后续 AI 功能提供统一的厂商和模型来源；
8. 支持流式对话与非对话型任务调用。

本需求第一版统一采用 **OpenAI 兼容协议** 接入，不接入非 OpenAI 兼容协议的厂商。

---

## 3. 目标与非目标

### 3.1 目标

本次需求目标如下：

- 在设置中新增 AI 供应商配置模块；
- 将 AI 供应商模块作为产品级通用 AI 基础设施；
- 内置常见 OpenAI 兼容厂商模板；
- 支持新增自定义厂商；
- 支持厂商下配置多个模型；
- 支持为厂商设置默认模型；
- 支持设置全局默认供应商；
- 支持模型级 `thinking` 开关；
- 在对话页面中按厂商和模型发起流式对话；
- 支持产品内非对话型 AI 调用场景复用该配置能力；
- 在选择供应商时自动带出默认模型，并允许用户手动切换；
- 当用户未配置可用 AI 时，为依赖 AI 的功能提供前置引导。

### 3.2 非目标

第一版明确不包含以下内容：

- Gemini 等非 OpenAI 兼容协议厂商接入；
- 不同厂商官方私有协议适配；
- 自动从供应商远程拉取模型列表；
- 多 API Key 轮询、权重或故障切换；
- 配额、计费、限流监控；
- 高级参数配置（如 temperature、top_p、max_tokens、context window 等）；
- 独立“测试模型”或“测试连接”按钮；
- 多轮会话管理、会话持久化策略细化；
- embedding、rerank、图像生成、语音等非文本对话能力。

## 4. 适用范围

### 4.1 设置页

新增 AI 供应商模块，用于统一管理厂商配置与模型配置。

### 4.2 对话页

对话页基于 AI 供应商模块中启用的厂商和模型进行选择与调用，支持流式输出。

### 4.3 简历生成功能

后续 AI 自动生成简历项目经历等任务型功能，应复用 AI 供应商模块中的可用供应商、默认供应商与默认模型配置。

### 4.4 其他 AI 功能

后续产品内新增的 AI 能力，应统一复用本模块提供的配置能力与调用能力，避免每个功能重复维护独立 AI 设置。

## 5. 核心概念定义

### 5.1 AI 供应商

AI 供应商是指一个可调用的 AI 服务端点，统一按 OpenAI 兼容协议接入。

每个供应商至少包含：

- 供应商名称
- 供应商类型（内置 / 自定义）
- Base URL
- API Key
- 启用状态
- 模型列表

### 5.2 模型

模型是供应商下的具体可调用能力单位。

每个模型至少包含：

- 模型名称
- 是否默认模型
- 是否启用
- 是否开启 `thinking`

### 5.3 默认模型

默认模型指某个供应商下在对话页面被自动选中的模型。

约束：

- 每个供应商最多只能有 1 个默认模型；
- 对话页面切换供应商时，系统自动带出该供应商默认模型；
- 用户仍可手动切换到同供应商下其他已启用模型。

### 5.4 全局默认供应商

全局默认供应商指产品内依赖 AI 的功能在未显式指定供应商时默认使用的供应商。

约束：

- 全局只能存在 1 个默认供应商；
- 默认供应商必须处于启用状态；
- 默认供应商下应存在至少 1 个可用默认模型；
- 对话页、简历生成等功能在未手动选择时，优先使用全局默认供应商。

### 5.5 Thinking

`thinking` 为模型级布尔配置项，用于表示该模型在请求时是否启用推理/思考能力。

说明：

- `thinking` 只在兼容能力的服务端生效；
- 若某兼容服务不支持该字段，第一版允许服务端忽略，不额外做复杂适配策略。

### 5.6 通用 AI 调用能力

本模块除服务对话页外，还需为产品内其他 AI 功能提供统一调用基础。

调用能力至少应覆盖两类场景：

- 流式对话型调用；
- 非对话型任务调用，例如基于项目上下文生成简历项目经历、总结内容、改写文本等。

## 6. 内置厂商范围

第一版内置以下厂商模板：

- 百炼 / 通义千问
- DeepSeek
- OpenAI
- Ollama
- Moonshot AI

说明：

- 内置厂商本质上是预置模板；
- 模板应提供默认名称和建议的 Base URL；
- 用户可修改名称、Base URL、API Key 与模型列表；
- 若厂商使用场景允许，API Key 可为空（例如部分本地 Ollama 场景）。

## 7. 用户角色与使用场景

### 7.1 用户角色

- 希望在一个应用中管理多个 AI 厂商的开发者
- 希望快速切换不同模型进行对话测试与日常使用的用户

### 7.2 典型使用场景

#### 场景一：新增 OpenAI 厂商并使用默认模型对话

1. 用户进入设置页；
2. 在 AI 供应商模块中新增 OpenAI 厂商；
3. 填写 Base URL 和 API Key；
4. 新增模型 `gpt-4o` 并设置为默认模型；
5. 进入对话页；
6. 选择该供应商后自动带出 `gpt-4o`；
7. 用户输入消息并获得流式回复。

#### 场景二：一个供应商配置多个模型

1. 用户为 DeepSeek 配置 `deepseek-chat` 与 `deepseek-reasoner`；
2. 将 `deepseek-chat` 设为默认模型；
3. 在对话页选择 DeepSeek 时自动使用默认模型；
4. 用户也可以手动切换到 `deepseek-reasoner` 再发起对话。

#### 场景三：自定义兼容 OpenAI 的厂商

1. 用户新增一个自定义厂商；
2. 填写自定义名称、Base URL 和 API Key；
3. 添加一个或多个模型；
4. 在对话页中像内置厂商一样正常使用。

## 8. 功能需求

## 8.1 设置页新增 AI 供应商模块

### 8.1.1 模块入口

在设置页中新增“AI 供应商”模块或同级菜单。

### 8.1.2 模块列表展示

AI 供应商列表应至少展示以下信息：

- 供应商名称
- 厂商类型（内置 / 自定义）
- Base URL
- API Key 状态（已配置 / 未配置）
- 是否为全局默认供应商
- 默认模型名称
- 启用状态
- 模型数量

### 8.1.3 支持操作

每个供应商需要支持以下操作：

- 新增
- 编辑
- 删除
- 启用 / 禁用
- 设为全局默认供应商
- 管理模型

---

## 8.2 新增/编辑供应商

### 8.2.1 供应商字段

新增或编辑供应商时，至少包含以下字段：

- 供应商名称
- 厂商类型
- Base URL
- API Key
- 是否启用
- 是否为全局默认供应商

### 8.2.2 厂商类型

厂商类型包括：

- 内置厂商模板
- 自定义厂商

### 8.2.3 内置厂商行为

选择内置厂商时：

- 自动带出默认名称；
- 自动带出建议 Base URL；
- 允许用户修改名称和 Base URL；
- 用户自行填写 API Key。

### 8.2.4 自定义厂商行为

选择自定义厂商时：

- 用户手动填写供应商名称；
- 用户手动填写 Base URL；
- 用户手动填写 API Key；
- 约束为 OpenAI 兼容协议。

### 8.2.5 API Key 展示规则

建议按以下规则实现：

- 已保存 API Key 不明文展示；
- 列表页仅展示“已配置”状态；
- 编辑时若用户未重新输入，则保留原值；
- 重新输入后覆盖旧值。

### 8.2.6 存储与安全原则

- AI 供应商配置默认保存在本地；
- 第一版不要求引入云端同步；
- 产品不应默认将用户 API Key 上传到产品云端；
- 对外展示时不应明文暴露 API Key；
- 未配置可用供应商时，应明确提示用户先完成 AI 配置。

---

## 8.3 模型管理

### 8.3.1 模型列表

每个供应商下支持维护多个模型。

模型列表应至少展示：

- 模型名称
- 是否默认
- 是否启用
- 是否开启 `thinking`

### 8.3.2 模型操作

模型需要支持：

- 新增模型
- 编辑模型
- 删除模型
- 设置默认模型
- 启用 / 禁用

### 8.3.3 模型字段

模型配置至少包含：

- 模型名称
- 是否默认模型
- 是否启用
- 是否开启 `thinking`

### 8.3.4 默认模型约束

- 同一供应商下只能存在一个默认模型；
- 若用户将某模型设置为默认，则同供应商下原默认模型自动取消默认；
- 默认模型建议必须是启用状态；
- 若默认模型被禁用或删除，系统应要求用户重新指定默认模型，或在有其他可用模型时自动选取一个可用模型作为默认模型。

---

## 8.4 产品级 AI 能力集成

### 8.4.1 对话页集成

对话页需要支持：

- 选择供应商
- 选择模型
- 输入用户消息
- 接收并展示流式响应

### 8.4.2 对话页默认带出规则

- 当用户选择某个供应商时，系统自动选中其默认模型；
- 若用户手动切换模型，则当前发送消息按用户选中的模型执行；
- 只能展示并使用已启用的供应商与模型；
- 当页面未显式指定供应商时，可优先使用全局默认供应商及其默认模型。

### 8.4.3 非对话型功能集成

后续类似简历生成、项目总结、内容改写等产品内 AI 能力，需复用本模块中的配置。

集成规则如下：

- 当功能页未显式指定模型时，默认使用全局默认供应商及其默认模型；
- 当用户未配置任何可用供应商或模型时，功能页需要提供前往 AI 设置页的引导；
- 非对话型功能调用同样统一走 OpenAI 兼容协议调用链路；
- 非对话型调用不强依赖流式返回，但调用层应允许统一复用同一套配置能力。

### 8.4.4 请求方式

第一版统一按 OpenAI 兼容协议调用，需同时支持：

- 流式对话请求；
- 非流式或任务型文本生成请求。

### 8.4.5 聊天验证方式

第一版不要求在设置页提供独立“测试连接”或“测试模型”按钮。

模型是否可用可通过用户在对话页中的真实聊天行为验证，后续其他任务型页面也可作为间接验证场景。

## 9. 页面与交互建议

## 9.1 设置页 - AI 供应商模块

建议采用“供应商列表 + 模型管理入口”的方式。

### 建议交互结构

- 左侧或上方：AI 供应商模块入口；
- 主区域：供应商列表；
- 列表操作：新增供应商；
- 单个供应商操作：编辑、删除、启用/禁用、管理模型。

### 新增/编辑供应商建议交互

可采用弹窗或抽屉形式，包含：

- 厂商类型选择
- 名称输入
- Base URL 输入
- API Key 输入
- 启用开关

### 管理模型建议交互

可在供应商卡片内展开，也可进入独立的模型管理弹窗。

模型表单包含：

- 模型名称
- 默认模型开关
- 启用开关
- thinking 开关

---

## 9.2 对话页

建议包含以下区域：

- 供应商选择器
- 模型选择器
- 消息输入区
- 消息列表区
- 流式响应展示区

### 交互规则

- 初次进入对话页时，若未手动指定，可默认选择全局默认供应商及其默认模型；
- 用户切换供应商后自动刷新模型列表；
- 若当前模型不可用，需要自动回退到该供应商默认模型；
- 发送消息时，按当前供应商和模型配置发起请求；
- 响应以流式方式逐步渲染。

---

## 9.3 其他 AI 功能页的前置引导

对于简历生成等依赖 AI 的功能页，建议遵循以下交互规则：

- 若系统不存在已启用的可用供应商或模型，则页面展示“请先完成 AI 配置”的提示；
- 提供跳转到 AI 设置页的快捷入口；
- 若系统存在全局默认供应商和默认模型，则功能页可直接使用，无需用户重复选择；
- 若页面允许高级用户切换模型，则切换范围仅限于已启用供应商和模型。

## 11. 规则与约束

### 11.1 协议约束

- 第一版只支持 OpenAI 兼容协议；
- 非兼容协议厂商不纳入本期范围；
- 所有 AI 请求统一走兼容 OpenAI 的调用链路；
- 调用层需同时支持流式对话与任务型文本生成场景。

### 11.2 供应商约束

- 供应商名称允许修改；
- Base URL 必填；
- 除特定允许为空的场景外，API Key 建议填写；
- 禁用状态的供应商不可在对话页或其他 AI 功能页使用；
- 全局只能存在 1 个默认供应商；
- 默认供应商必须为启用状态。

### 11.3 模型约束

- 模型名称必填；
- 同一供应商下仅允许一个默认模型；
- 禁用状态的模型不可在对话页或其他 AI 功能页使用；
- `thinking` 为模型级配置。

### 11.4 前置引导约束

- 当产品内 AI 功能页检测不到可用供应商或模型时，必须提供前往 AI 设置页的引导；
- 依赖 AI 的功能不应在无配置状态下直接报空白错误或静默失败。

## 12. 验收标准

### 12.1 设置页验收标准

- 设置页中存在 AI 供应商模块入口；
- 用户可以新增内置厂商模板；
- 用户可以新增自定义厂商；
- 用户可以编辑、删除、启用/禁用供应商；
- 用户可以将某个供应商设为全局默认供应商；
- 用户可以为供应商配置多个模型；
- 用户可以设置默认模型；
- 用户可以配置模型级 `thinking` 开关；
- 已保存 API Key 不在列表中明文显示。

### 12.2 对话页验收标准

- 对话页可选择已启用的供应商；
- 未手动指定时可自动带出全局默认供应商及其默认模型；
- 选择供应商后可自动带出其默认模型；
- 用户可切换到该供应商下其他已启用模型；
- 对话请求按 OpenAI 兼容协议发送；
- 对话响应支持流式输出；
- 禁用的供应商或模型不会出现在可选项中。

### 12.3 自定义厂商验收标准

- 用户可新增一个自定义厂商；
- 自定义厂商可配置 Base URL、API Key 和模型；
- 自定义厂商在对话页和其他 AI 功能页的使用方式与内置厂商一致。

### 12.4 其他 AI 功能集成验收标准

- 简历生成等任务型 AI 功能可以复用 AI 供应商模块中的可用配置；
- 当功能页未显式指定供应商和模型时，可默认使用全局默认供应商及默认模型；
- 当系统无可用 AI 配置时，功能页会提示用户先完成 AI 配置；
- 功能页提供跳转到 AI 设置页的入口；
- 非对话型 AI 功能调用与对话页调用共用统一的 OpenAI 兼容配置来源。

## 13. 风险与注意事项

### 13.1 Thinking 兼容性

虽然界面统一提供 `thinking` 开关，但不同 OpenAI 兼容服务对该字段的支持程度不同。

处理建议：

- 配置层统一保留该字段；
- 调用层按兼容能力透传；
- 对不支持的服务允许忽略，不阻断基础对话能力。

### 13.2 Base URL 差异

不同供应商的 OpenAI 兼容地址路径可能存在差异。

处理建议：

- 内置模板提供推荐地址；
- 允许用户手动修改；
- 请求层避免对 URL 做过强的硬编码假设。

### 13.3 本地 Ollama 场景

Ollama 常见为本地部署，可能不需要 API Key。

处理建议：

- 允许特定场景 API Key 为空；
- 在表单提示中说明该行为。

### 13.4 与 Issue #21 的关系

本 PRD 对应 Issue #31 的直接实现需求，同时也是 Issue #21 的前置基础能力文档。

支撑关系如下：

- Issue #21 中的 AI 简历生成能力不应单独维护模型配置；
- 简历生成功能应直接复用本模块中的全局默认供应商、默认模型与调用链路；
- 若用户尚未完成 AI 配置，则简历生成功能应先引导用户进入本模块完成配置；
- 后续其他 AI 功能也应沿用同一设计原则。
